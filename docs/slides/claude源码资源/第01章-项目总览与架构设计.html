WHAT IS IT

三个核心特点

不是简单的命令行包装器，而是能自主读写文件、执行命令、调用工具的完整系统

🖥️
CLI 优先的 UI
界面完全跑在终端，使用自研 Ink 框架（基于 React），将组件树渲染为 ANSI 转义序列。颜色、进度条、提示框——背后都是 useState 与重渲染
React → ANSI 转义序列
⚡
AsyncGenerator 数据流
核心查询循环不是 Promise，而是 AsyncGenerator。每轮 API 调用、每次工具执行的中间结果都以 yield 形式流出，支持任意时刻订阅、暂停或中止
async function* submitMessage()
🏗️
Feature Flag 编译时分叉
feature() 来自 bun:bundle，在 Bun 打包时求值。未启用的代码分支被彻底裁剪——内外版本共享源码，却生成完全不同的二进制

CORE ARCHITECTURE

AsyncGenerator 管道

一次完整用户问答轮次（Turn）的数据流路径 — 点击逐步展开

①
用户击键 → stdin 捕获
Ink 框架通过 process.stdin 捕获原始键盘事件，经 hooks/useTextInput 处理后触发 REPL 组件的状态更新
↓
②
processUserInput — 路由分发
识别输入类型，斜杠命令与自然语言走不同路径 utils/processUserInput/processUserInput.ts
↓
路径 A — 斜杠命令
commands/ 处理器
/clear · /compact · /commit · 70+ 命令
路径 B — 自然语言 ↓
query() 核心循环
src/query.ts · AsyncGenerator
↓
③
Claude API → 流式事件
组装 System Prompt（含 CLAUDE.md），发送请求，AsyncGenerator 逐条 yield：assistant_message · stream_event · compact_boundary
↓
④
工具调用 → BashTool · FileEditTool · AgentTool
解析 tool_use 块，构建 ToolUseContext（含 AbortController · 权限检查 · UI 更新通道），执行工具后 ToolResult 包装为 user 消息 重入 query() 循环
↓
⑤
end_turn → QueryEngine yield → Ink 渲染
stop_reason='end_turn' 后，QueryEngine.submitMessage() yield 出 SDKMessage，REPL 经 useLogMessages Hook 订阅更新，触发 Ink 重渲终端输出
DATA FLOW

六阶段数据流

从用户击键到终端渲染的完整旅程，每阶段职责清晰

STAGE 1
用户输入捕获
Ink 通过 process.stdin 捕获键盘事件，hooks/useTextInput 处理后更新 REPL 组件状态
STAGE 2
输入预处理
processUserInput 识别斜杠命令并路由到 commands/，普通输入打包为 UserMessage 继续
STAGE 3
query() 核心循环
组装 System Prompt（含 CLAUDE.md 内容），发给 Claude API，处理自动压缩与 Token 预算超限
STAGE 4
工具执行
找到工具（findToolByName），构建 ToolUseContext，调用 tool.call()，执行 Bash · FileEdit · Agent · MCP
STAGE 5
结果归集与递归
ToolResult 包装为 user 消息重入 query()，直到 end_turn · Token 耗尽 · 用户中断 · maxTurns 达到
STAGE 6
Ink 渲染输出
QueryEngine yield SDKMessage → useLogMessages Hook 订阅 → React 状态更新 → Ink 重渲终端

  • projectSettings — 项目级
  • localSettings — 本地覆盖
  • flagSettings — 命令行，最高
  • policySettings — 企业锁定，防覆盖
🚩
Feature Flag（两种机制）
  • 编译时：bun:bundle 的 feature()
  • 未启用分支物理删除于产物
  • 外部版本 SleepTool 根本不存在
  • 运行时：GrowthBook 动态开关
  • A/B 测试与灰度发布

全局单例状态的设计哲学

src/bootstrap/state.ts:31 · 259 · 428-429 — 整个代码库注释最密集的文件
全局状态是必要的恶，不是便利的工具
约 60 个字段，每个字段都有清晰的注释说明其存在理由。三条重复出现的强调性警告，体现了作者对状态膨胀的高度警惕
函数式 Setter 保证状态可追踪
没有 STATE.field = value 这样的直接赋值暴露在外。所有修改通过 setAppState(f: prev → AppState) 进行，兼容 React 不可变状态模型，使变更可追踪
